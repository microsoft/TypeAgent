# Revisit triggers

A consolidated index of v1 design decisions that were closed with an
explicit condition under which they should be reopened. Each row names
the decision, the chosen v1 position, the trigger (the observation in
practice that would justify reopening), and the source where the
analysis lives.

This is a living index. When a decision is recorded with a deferred
alternative and a "trigger to revisit" clause, add a row here so the
condition is discoverable without reading every §8.x subsection or
decision record.

## Cross-cutting triggers

Two observations would justify reopening many of the rows below at
once, not just one. They sit above the per-decision table because
they are about the IR's fitness for the populations §1.1 prioritizes,
not about any single design choice:

1. **DSL cannot lower cleanly to the IR.** The IR is designed as a
   compile target (§1.1.2 codegen as primary writer, §1.2 verbose by
   design). If a DSL repeatedly cannot lower a common construct
   without faking IR shapes, side tables, or post-emission rewrites,
   the IR is failing its primary writer. The reopen is not "fix this
   one row"; it is "audit which rows are forcing the friction."
   Decisions most likely to come up: §8.15 hide-by-default `bind`,
   §8.17 pure SSA per namespace, §8.5 centralized `iterateState`,
   §8.16 task schema source. The audit may also surface a missing
   post-v1 mechanism (e.g., edge-scoped bind, block scope) that
   would close the gap additively.
2. **Engine performance suffers because of the lowering.** The IR's
   verbose-by-design tax assumes the engine pays no runtime cost for
   it (validation cost is one-time at load; runtime walks the
   compiled form). If profiling shows the lowering itself is the
   bottleneck (e.g., per-reference namespace dispatch, snapshot
   construction at every iteration boundary, schema re-validation hot in the
   inner loop), then v1 is taxing the dominant population. The
   reopen targets whichever rows the profiler points at; candidates
   include §8.1 type encoding, §8.3 branch model, §8.16 schema
   restatement, and the snapshot-read rule in §3.7.1.

Either trigger reopens decisions in clusters. The per-decision rows
below are still the correct catalog for narrow reopenings driven by
their specific triggers.

## Adjacent: post-v1 features (not "revisit", but "add when needed")

[ir-v0.1.md §2.2](./ir-v0.1.md) lists features deliberately deferred from
v1 (sub-workflow calls, capability declarations, parallelism
annotations, block scope, nested loops, ...). Those are additive
post-v1 work, not reopenings of a closed v1 decision, and live in their
own table. Use this revisit-triggers index for "the v1 decision was X
and may need to flip to Y if Z is observed."
