# How the wiki is structured

This page explains the content model behind the wiki so contributors know
**where to put things** and **why**. The guiding goal is: _one piece of
knowledge, one home_ — no duplicated content to keep in sync.

## The docset is `ts/docs` itself

DocFX builds from `ts/docs` directly, so almost everything is a **native file**
in this tree and needs no copying:

- **Conceptual content** — the [Overview](../index.md), this contributor
  guide, and the [templates](../templates/package-page.md) — is authored here.
- **Architecture deep dives** live under `ts/docs/architecture/**` and are edited
  in place; the build does not copy them.
- The former separate **Design** tree has been folded into
  `architecture/workflows/` (the "Workflow system" group).

> The `ts/docs/plans/**` directory holds repo planning docs but is **excluded
> from the wiki build** (see the `plans/**` exclude in `docfx.json`); it is not
> part of the published wiki.

The **only** content that lives outside `ts/docs` is the package/agent
documentation under `ts/packages/**`. That is the one thing the build crawls and
copies in.

## Staged content: package & agent docs

[`scripts/build-wiki.mjs`](https://github.com/microsoft/TypeAgent/blob/main/ts/docs/scripts/build-wiki.mjs)
**stages** each package's documentation under the docset before the build,
rewriting every relative link so it still resolves (an in-wiki link when the
target is also staged, an absolute GitHub URL otherwise) and prepending a short
"source of truth" banner. The staged files are git-ignored:

| Canonical source                 | Staged under                                | As                               |
| -------------------------------- | ------------------------------------------- | -------------------------------- |
| package root `README.md`         | `packages/<pkg>/overview.md`                | **Overview** (primary page)      |
| package root `README.AUTOGEN.md` | `packages/<pkg>/generated.md`               | **Generated README** (secondary) |
| any **other** root-level `*.md`  | `packages/<pkg>/<name>.md` (name preserved) | a sibling page                   |
| the package's `docs/` directory  | `packages/<pkg>/docs/**` (mirrored)         | a nested "docs" subtree          |
| `ts/packages/agents/**`          | `agents/<agent>/...`                        | same rules as packages           |

So **every markdown file at a package's root** is published (only `README.md`
and `README.AUTOGEN.md` are renamed), and if a package has a `docs/` folder its
whole tree (markdown + images) is mirrored. Put structured package documentation
in `docs/` so it lands in a tidy sub-section.

Because the staged copies are regenerated on every build, **edit the file in the
package** — never a staged copy. The `WIKI_PAGE_POLICY` environment variable
(`both` (default), `overview`, `generated`) controls whether a package shows both
README pages or just one.

## Navigation: generated TOCs

Each section has a `toc.yml` that drives its left-hand navigation:

- **Hand-written TOCs** — `overview/` and `contributing/` have stable,
  curated `toc.yml` files.
- **Generated TOCs** — `architecture/`, `packages/`, and `agents/` have
  `toc.yml` files produced by `build-wiki.mjs`. The architecture toc is built
  from the `ARCH_GROUPS` taxonomy in the script (which groups docs into
  _Core routing_, _Collision resolution_, _Browser_, _Workflow system_, …); the
  others mirror the file system. Generated files start with an
  `# <auto-generated>` header — **do not edit them by hand**; rerun the script.

```bash
# Stage package docs + regenerate the generated TOCs.
node ts/docs/scripts/build-wiki.mjs

# CI-friendly check: non-zero exit if any architecture doc is not assigned to
# a group in the taxonomy.
node ts/docs/scripts/build-wiki.mjs --check
```

## Directory layout

```
ts/docs/
  docfx.json            DocFX docset configuration
  toc.yml               top-level navigation (the sections above)
  index.md              wiki landing page
  overview/             native: what / principles / getting-started / glossary
  architecture/         native deep dives grouped into sub-dirs; generated toc.yml
    core/  collision/  agents/  browser/  workflows/  doc-pipeline/
  plans/                repo planning docs — EXCLUDED from the wiki build
  packages/  agents/    curated index.md; the rest staged from ts/packages + generated toc.yml
  contributing/         this guide
  templates/            page templates
  scripts/              build-wiki.mjs
  media/                wiki-owned images
```

## How new content flows in

| Add a…                      | Edit / create                                     | Then                                                                                                  |
| --------------------------- | ------------------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| Curated wiki page           | a `.md` under the relevant `ts/docs/<section>/`   | add one line to that section's `toc.yml`                                                              |
| Architecture / workflow doc | a `.md` under `ts/docs/architecture/<group>/`     | rerun the script; if it is a new file at the architecture root, assign it to a group in `ARCH_GROUPS` |
| Package / agent             | the package's `README.md` (+ `README.AUTOGEN.md`) | rerun the script (or let doc-autogen do it)                                                           |

The [doc-autogen pipeline](./doc-autogen.md) is expected to run the script as
part of its job, so once it is back online the package/agent navigation
refreshes automatically and contributors rarely need to run it by hand.
